# SM90 true-NVFP4 on-chip sparse grouped GEMM design

## Goal

Replace the current two-dense-WGMMA SM90 NVFP4 grouped GEMM fast path with a
path that can approach one dense FP8 GEMM worth of tensor-core work while
preserving the strict Cupra elementwise contract.

The primary performance target is at least 80% **useful Cupra utilization** on
the steady-state large-M shape `8192x4096x7168/E8`. This is intentionally not
the current "two FP8 terms combined reach 81% physical utilization" metric.
The M=128 and M=512 task shapes must also improve materially from the Iter 14
measurements of 0.3877 ms and 0.1627 ms; a first completion gate is at least a
10% latency reduction on each, followed by continued tuning if profiling still
shows avoidable overhead.

## Non-negotiable storage invariant

Persistent weights remain genuine NVFP4:

- packed E2M1 FP4 codes, one byte per two weights;
- one E4M3 block scale per 16 weights;
- one FP32 global scale per expert.

`prepare()` may only perform a lossless byte-preserving permutation. Its output
must have the same dtypes and exactly the same number of weight and scale bytes
as its input. It must not create an FP8/BF16 copy, duplicate a weight payload,
or create correction values or indices. The existing layout
`[E,K/16,N,8]` for packed bytes and `[E,K/16,N]` for block scales satisfies
this invariant.

H20 has no native FP4 MMA instruction. FP8 values, sparse metadata, and
correction data may therefore exist transiently in registers or CTA shared
memory, but never as a persistent/global weight representation.

## Considered approaches

### 1. Two on-chip 2:4 sparse terms plus an on-chip scalar tail (selected)

Decode each packed FP4 K group directly into two complementary 2:4 E4M3
operands in shared memory. Hopper executes each sparse WGMMA at twice dense
throughput, so two sparse terms have the tensor-core issue cost of one dense
term. A very small residual tail is accumulated on CUDA cores from the same
packed FP4 source while WGMMA is in flight.

This is the only evaluated approach that both preserves true packed FP4 and
has a theoretical path to 80% useful utilization.

### 2. Dense primary plus sparse residual

Decode a dense primary term and prune the residual to 2:4. Numerical prototypes
can satisfy the tolerance, but one dense plus one sparse WGMMA costs the
equivalent of 1.5 dense terms. Its pre-overhead useful ceiling is 66.7% of the
dense FP8 peak, so it cannot meet the target.

### 3. One dense primary plus scalar residual

A single dense primary is attractive structurally, but the largest shape needs
about 15.1% of weights corrected (`~1081` correction terms per weight row) to
pass. That scalar work is too large to hide and is rejected.

The current exact two-dense-term implementation remains a correctness fallback,
but its useful ceiling is 50% before overhead.

## Numerical decomposition

For one normalized NVFP4 value

```text
x = E2M1(code) * E4M3(block_scale) / 8
```

form two E4M3 candidates:

```text
q0 = round_e4m3(x)
q1 = round_e4m3(x / 3)
```

For every consecutive four K positions, compare the squared reconstruction
error of `q0` and `3*q1`. Select exactly two positions for the `q1` operand and
the complementary two for the `q0` operand. Both generated operands are then
valid Hopper 2:4 sparse matrices, and their reconstructed value is

```text
x_hat[k] = q0[k]       for term-0 positions
x_hat[k] = 3 * q1[k]   for term-1 positions
```

The sparse values and their metadata are generated by the producer directly
from packed FP4 bytes and the scale LUT. They live only in the current shared
memory pipeline stage.

## Deterministic residual correction without a table

The residual `r = x - x_hat` has discrete error levels. Correct all values with
`abs(r) > 3`, and correct one deterministic quarter of values with
`abs(r) > 1.5` using

```text
h = (k * 0x9E3779B1) ^ (n * 0x85EBCA77) ^ (expert * 0xC2B2AE3D)
selected = ((h >> 16) & 3) == 0
```

The rule is recomputed on chip; no correction index/value table exists.
Measured correction counts are approximately 8-9 terms per full weight row at
K=2048 and 26-27 terms at K=7168. Across K64 stages, the mean is 0.24 terms per
row per stage. A full 64-bit mask plus transient half residual values is used
for each row/stage, so correctness never depends on a fixed compact-list
capacity.

The numerical prototype, using Cupra's actual NVFP4 quantizer and tolerance,
produced zero violations for all six task shapes at seed 0 and zero violations
for the largest shape at seeds 0 through 4.

## Kernel architecture

Sparse WGMMA requires the sparse operand to be matrix A. Compute the transposed
form

```text
C.T = W * activation.T
```

with a logical tile of 64 weight rows, a tunable token width, and K64:

1. A 128-thread producer warpgroup loads packed FP4 bytes, block scales, and
   activation bytes for the next stage.
2. The producer uses LUT/PRMT decoding to generate the two compressed E4M3
   sparse operands, both metadata streams, and the residual mask/values.
3. A 128-thread consumer warpgroup issues the two sparse WGMMA streams into
   separate FP32 accumulator fragments.
4. While tensor-core work is asynchronous, the consumer accumulates the rare
   residual terms from the activation tile already resident in shared memory.
5. The epilogue combines `acc0 + 3*acc1 + correction`, applies
   `8 * activation_scale * expert_global_scale`, transposes the tile through
   shared memory, and writes coalesced BF16 output.

The large path starts with token width 128. Two pipeline stages use far less
shared memory than the current pair of dense decoded operands, leaving room for
the correction mask and transposed epilogue tile. Register pressure, especially
from the two WGMMA fragments and correction fragment, is a measured tuning
constraint; token width 256 is not assumed viable.

## Small-M policy

Small M must not inherit a token-width-128 padding penalty for every expert.
Compile token-width 32, 64, and 128 variants:

- M=128 / many short expert segments starts with width 32 and a simple
  one-warpgroup schedule.
- M=512 starts with width 64.
- steady-state large M uses width 128 and producer/consumer specialization.

Dispatch uses expert-tail-aware tile estimates. The current exact kernel stays
available as a fallback, and the benchmark decides the crossover rather than a
hard-coded assumption.

## Correctness and safety

- The canonical public API remains valid and consumes canonical packed FP4.
- The fast layout is byte-preserving and must be checked by an explicit byte
  count/dtype invariant.
- Empty experts, unaligned expert token tails, and every K block are predicated.
- Residual storage uses a full mask rather than a bounded list, avoiding silent
  overflow on adversarial weights.
- All generated FP8/sparse state is CTA-local and is discarded after use.

## Verification gates

Each implementation iteration must:

1. compile on H20 and pass a focused shape before performance is considered;
2. pass Cupra precision for all six shapes without changing tolerance;
3. retain the two canonical edge tests (empty expert/unaligned M and multi-K);
4. run memcheck and synccheck on the simple and warp-specialized paths;
5. report Cupra useful TFLOPS/utilization, not only physical WGMMA work;
6. measure M=128 and M=512 independently with jitter; and
7. verify that prepared weight/scale bytes equal the canonical input bytes.

The existing exact implementation is kept until the new path clears every
correctness and safety gate and is faster at its selected dispatch points.
